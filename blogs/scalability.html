Scalability

As the system grows (in data volume, traffic volume, or complexity), there should be reasonable ways of dealing with that growth. A service is said to be scalable if when we increase the resources in a system, it results in increased performance in a manner proportional to resources added.

Performance vs Scalability üìà

Performance and scalability are two related but distinct aspects of a system. Another way to look at performance vs scalability:

  • If you have a performance problem, your system is slow for a single user.
  • If you have a scalability problem, your system is fast for a single user but slow under heavy load.

Performance

Factors such as network latency, bandwidth, and other elements also affect performance. Everything works on trade-offs. For instance, in order to increase performance, we could optimize the code, database, or network, which can improve performance, but it can also reduce flexibility, modularity, or maintainability.

Scalability

Scalability is the ability of the system to handle a large amount of work when the system is under heavy load by adding more resources to the problem. Scalability is not a 1D label; it means asking questions like, ‚Äúif a system grows in a particular way, what are our options for coping up with this growth without reducing the performance?‚Äù There are two ways of scaling basically, Vertical (Scaling up) and horizontal (Scaling out).

Basic Steps to Consider:

  1. Describe the load üìö ‚Äî What kind of load are we expecting? It can be anything, could be requests per second for a web server or read and write per second for a Database, to define what we expect and how we can design a system to handle this.
  2. Describe the performance ‚Äî Now that we have an increased load, what is the scope of performance? Would this load make an impact directly to the user interface, or would it be okay to be handled like a back office? Depending on various situations, decisions have to be considered and made.

Horizontal Scaling and Vertical Scaling

Vertical scaling üö¶ is generally used in monolithic architecture where the service (Existing) is scaled up by increasing CPU power, memory, or storage capacity to accommodate the increased demands of your application.

Horizontal scalability üö•, or ‚Äúscaling out,‚Äù refers to the expansion of an application by adding more hardware or nodes to distribute the workload. This is achieved by adding physical or virtual machines, which work in parallel to increase processing power and capacity. Also, one important aspect to keep in mind with respect to Horizontal scalability is that our a system is not horizontally scalable if the code design does not allow multiple servers to respond to requests. A good example of a system that is horizontally scalable but not vertically scalable is a distributed NoSQL database like Cassandra or MongoDB.
